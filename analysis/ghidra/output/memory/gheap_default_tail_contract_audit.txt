AUDIT: gheap Default heap tail-adoption safety contract
======================================================================

Context:
  gheap adopts the unused tail of the vanilla Default heap after
  skipping the first 16MB. Runtime logs show adoption later disables
  only because live/free counters changed in the front zone:
    live 1546KB->1543KB, free 0->2
  while bump/high-water and first/last physical blocks stayed below
  the adopted tail. This script checks which fields are real tail-use
  evidence and which are normal front-zone allocator counters.

Known large-heap fields from earlier audits:
  +0x14 reserve capacity
  +0x18 initial commit size
  +0x1c committed limit
  +0x24 bump/end offset
  +0x28 high-water offset
  +0x2c current live bytes
  +0x30 reservation base
  +0x34 physical block count
  +0x38 unknown/history
  +0x3c first/last physical block pointer from prior runtime naming
  +0x40 free block count
  +0x44..+0x2043 free-list buckets
  +0x2048/+0x204c large free-list sentinel/head

Questions to answer from output:
  1. Does a changed +0x2c/+0x34/+0x40 imply tail use?
  2. Are +0x24/+0x28 monotonic upper bounds for all created blocks?
  3. Is +0x3c enough to prove every physical block remains below tail?
  4. Can gheap keep adopting tail while front-zone live/free counters move?
