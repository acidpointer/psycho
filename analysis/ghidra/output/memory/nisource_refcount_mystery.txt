NISOURCETEXTURE REFCOUNT MYSTERY
======================================================================

Hypothesis 1: QueuedTexture+0x30 NiPointer is never set (stays NULL)
  -> crash is NOT from stale NiSourceTexture at +0x30
  -> crash is from QueuedTexture ITSELF being freed/recycled

Hypothesis 2: NiSourceTexture freed via path ignoring refcount
  -> need to find all free paths for NiSourceTexture

Hypothesis 3: The QUEUEDTEXTURE is what gets freed, not NiSourceTexture
  -> QueuedTexture is freed by one DecRef, main thread does second DecRef
  -> double-free on QueuedTexture itself

PART 1: DOUBLE DECREF ANALYSIS
======================================================================

From crash calltrace:
  0x0044DDA1 (DecRef) -- called from 0x0043BE3F
  0x0044DDA1 (DecRef) -- called from 0x00C3D38F area

If refcount starts at 1, first DecRef -> 0 -> destructor runs.

FUN_00c3c620 which processes more tasks... which call DecRef
on the SAME QueuedTexture AGAIN -> refcount goes to -1 -> 
(*vtable[0])(1) called on already-freed QueuedTexture -> CRASH
