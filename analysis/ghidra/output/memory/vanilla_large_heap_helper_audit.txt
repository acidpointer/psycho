AUDIT: vanilla Default/File heap helper layer
======================================================================

Goal: find a reliable live-allocation test before reclaiming old
Default/File heap reservations after gheap activation.

Known object fields from earlier audits:
  +0x10 granularity
  +0x14 reserve capacity
  +0x18 initial commit size
  +0x1c committed limit
  +0x24 bump/end offset
  +0x28 high-water offset
  +0x2c suspected currently-allocated bytes; verify here
  +0x30 reservation base
  +0x3c first physical block/list head
  +0x44..+0x2043 free-list buckets
  +0x2048 large/free-list head
  +0x2058/+0x205c bitmap/search helpers
  +0x20d8 critical section
  +0x20f0 debug/alternate mode byte

Block format suspected from allocator:
  block+0x00 previous physical block
  block+0x04 size low 28 bits, flags high nibble
  block+0x08 user payload for allocated blocks
  block+0x08/+0x0c free-list links for free blocks

Questions to answer from this output:
  1. Does +0x2c decrement on free and increment on alloc?
  2. Does +0x24 return to 0 ever, or is it only bump/end?
  3. Is a fully free heap represented by one free block at +0x30?
  4. Does destructor 00aa7930 release backing storage and clear fields?
  5. Can we safely call vanilla release, or must we update fields manually?
