# FNV Object PBR Temporal Instability Audit

## Verdict

The visible object change is not evidence that global metallicness is switching
on and off. The installed OMV configuration has object metallicness `0.0`.

The regression is a hybrid shader-contract failure introduced after the NVR
object shader was ported:

1. OMV enabled NVR's commented-out screen-derivative `SpecularAA` experiment.
   This makes roughness depend on the current pixel footprint, so distance,
   projected size, mip selection, and camera orientation can change it.
2. OMV split the unified NVR BRDF and multiplied its GGX specular component by
   vanilla's distance fade. Vanilla applies that fade to a different, bounded
   gloss model. The first OMV curve compressed the fade into ten percent of the
   native interval; the follow-up linear curve exposed the unstable GGX lobe
   across the entire interval and made many more objects visibly change.
3. OMV changed NVR's dielectric-only object calls from metallicness `0` to the
   unmasked global object setting. This is not causing the captured run because
   the setting is zero, but it is an incorrect and dangerous source divergence.
4. OMV's shipped roughness scale is `0.82`, while NVR defaults to `1.0`. This
   makes common object surfaces glossier and amplifies the first two failures.

The immediate correction is to restore the source-proven NVR object behavior,
not invent another fade curve.

## Evidence Set

### Complete vanilla shader archive

`analysis/shaders_disasm/` contains all installed FNV SDP shader records:

- 16 packages;
- 15,791 records;
- 8,160 pixel shaders and 7,631 vertex shaders;
- 156 duplicate-named records retained with deterministic occurrence/offset
  suffixes;
- zero D3DX disassembly failures;
- manifest count equals `.dis` file count exactly.

The archive was generated by `analysis/disassemble_sdp_shader.c`. Each package
has a `_manifest.tsv` recording the original package offset and output file.

### Vanilla combined-specular object contract

The high-quality package's ordinary pair is:

- `analysis/shaders_disasm/shaderpackage019/SLS2012.vso.dis`;
- `analysis/shaders_disasm/shaderpackage019/SLS2017.pso.dis`.

The vertex shader:

- transforms the light direction into tangent space;
- constructs and transforms the half vector from `EyePosition`, object
  position, and `LightData[0].xyz`;
- copies `LightData[0].w` to `oT1.w`.

The pixel shader:

- samples the normal map and preserves its alpha;
- computes `pow(saturate(dot(normal, half)), c27.z)`;
- multiplies that lobe by normal-map alpha;
- applies the low-`NdotL` correction;
- saturates the per-light specular color;
- multiplies only that bounded specular result by `t1.w`;
- adds diffuse and ambient/fog separately.

The same semantic sequence exists in all installed packages for the ordinary
and high-light combined-specular rows. The relevant package-19 pairs are:

- `SLS2012.vso / SLS2017.pso`;
- `SLS2016.vso / SLS2023.pso`;
- `SLS2025.vso / SLS2034.pso`;
- `SLS2026.vso / SLS2035.pso`.

Across all 16 packages, every one of those four pixel rows applies `t1.w` to
the accumulated vanilla specular term. The exact register allocation differs
between quality packages, but the contract does not.

### What normal-map alpha means to vanilla

Vanilla treats normal-map alpha as specular strength. It does not use it as the
sole specular-lobe-width control. Material `glossPower` in `Toggles.z` controls
the exponent, while normal alpha scales the resulting lobe before saturation.

This distinction matters because changing normal alpha from amplitude to
roughness can turn an ordinary high-strength material into a nearly mirror-like
GGX material.

### Actual NVR object PBR design

The relevant source is:

- `.research/TESReloaded10-master/src/hlsl/NewVegas/Shaders/ObjectTemplate.hlsl`;
- `.research/TESReloaded10-master/src/hlsl/NewVegas/Shaders/Includes/Object.hlsl`;
- `.research/TESReloaded10-master/src/hlsl/NewVegas/Shaders/Includes/PBR.hlsl`;
- `.research/TESReloaded10-master/src/effects/PBR.cpp`;
- `.research/TESReloaded10-master/src/effects/PBR.h`.

NVR object PBR is not a metalness-mapped material system:

- every object BRDF helper call passes metallicness `0`;
- defaults explicitly describe object metallicness as unused because no metal
  maps exist;
- normal alpha is converted to roughness;
- object PBR is evaluated as one unified diffuse-plus-specular result;
- the ordinary object path does not apply `LightData[0].w` to PBR specular;
- the high-light path leaves a TODO about vanilla's attenuation but does not
  apply it;
- `SpecularAA` exists in the include, but its call in `ObjectTemplate.hlsl` is
  commented out behind old debug scaffolding;
- default roughness, light, ambient, and saturation scales are all `1.0`.

Therefore OMV's current object shader is not an NVR-source-equivalent port. It
combines NVR's BRDF with additional experiments that change temporal behavior.

## Why The Symptom Looks Like PBR Switching

OMV computes object roughness with a lower bound of `0.043`. In its GGX
distribution, peak `D` is approximately `1 / (PI * roughness^4)`. At the lower
bound this is about 93,000 before Fresnel, geometry, light color, and the rest
of the BRDF are applied. Vanilla saturates its specular light contribution;
OMV's GGX path does not apply an equivalent per-light bound.

The enabled `SpecularAA` adds screen-space normal variance to roughness and can
raise it toward roughly `sqrt(0.043^2 + 0.18)`, or `0.426`. At that roughness,
the same peak distribution is only about `9.7`. A surface moving between those
regimes can therefore change the peak distribution by thousands of times.

This explains the observed presentation:

- far/small object: larger screen-space normal derivatives, larger roughness,
  weak broad highlight, object appears dark;
- nearer/larger object: smaller derivatives, roughness falls, narrow intense
  highlight appears, object looks as if PBR switched on;
- camera rotation: quad footprint, mip choice, visible normal samples, and the
  view-dependent GGX lobe change, so the same issue appears without a large
  forward movement;
- compressed fade: highlight appears over a short distance band and looks like
  a blink;
- full linear fade: more objects occupy a partially enabled unstable specular
  state, matching the reported worse result.

The BRDF is naturally view dependent, but ordinary view dependence alone does
not justify a material-looking on/off transition. The extreme lobe, derivative
roughness, and mismatched distance fade make that dependence objectionable.

## Metallicness Finding

The installed runtime config is:

```toml
metallicness = 0.0
roughness_scale = 0.82
light_scale = 1.15
ambient_scale = 1.10
albedo_saturation = 1.02
```

So metallicness is ruled out as the active scalar behind this run. OMV's use of
global object metallicness still needs removal because NVR deliberately passes
zero and no object metalness mask exists.

## Corrective Change Set

### Phase 1: restore the NVR object contract

1. Remove both calls that apply `getObjectSpecularTransition` to combined PBR
   specular in `ObjectTemplate.hlsl`.
2. Remove the split `PBRLightingComponents` accumulation used only to apply the
   native fade. Restore unified NVR `PBR`/`PBRSun` accumulation.
3. Remove unconditional `SpecularAA` calls. Keep the helper unused for reference
   or remove it in cleanup; do not expose it as normal behavior.
4. Pass object metallicness `0` exactly as NVR does. Keep terrain metallicness
   separate because NVR terrain really uses it.
5. Restore object profile defaults to NVR-neutral `1.0` values. If tuned terrain
   values are desired, separate object and terrain profiles instead of sharing
   a single legacy profile.
6. Preserve OMV's safe-normalization guards because they only handle degenerate
   vectors and do not introduce screen-space state.
7. Preserve shader pair coverage, wrapper ownership, constant upload, sampler,
   fallback, and diagnostics contracts. This is not a feature disable.

Implementation status (2026-07-16): complete in source. The object shaders now
use unified NVR accumulation, contain no derivative roughness adjustment, do not
apply the native fade, and pass dielectric metallicness zero. Object and terrain
settings/constants are separate; legacy generic profile keys migrate to terrain
only, while object values start at neutral defaults. Shader cache keys include a
new material-contract revision. The i686 release DLL builds and every registered
PBR shader variant compiles through the runtime D3D compiler. In-game motion
validation remains required because static and compiler checks cannot reproduce
the reported scene transitions.

### Phase 2: ordinary playtest validation

Static closure now covers the complete object registry, native special-row
attenuation ABI, sampler masks, light-count gates, finite/continuous vector and
attenuation math, bounded-BRDF fade behavior, every real shader compilation,
and family-specific GPU budgets. No diagnostic gameplay run is required.

Test the same identified rocks/statics with four motions in one ordinary
playtest:

1. rotate in place while keeping the object visible;
2. walk directly toward and away from it;
3. strafe at approximately constant distance;
4. cross the native specular-fade range and object LOD boundary.

Keep detailed PBR draw telemetry disabled. Verify that visual response remains
continuous across all four motions, projected shadows remain native, the
special point-light falloff matches vanilla geometry, and frame rate does not
regress materially. Static tests prove the row/resource/equation contracts;
this playtest is only the final pixel and driver acceptance gate.

### Phase 3: material-faithful enhanced PBR, only if still needed

If source-equivalent NVR remains too shiny, do not re-enable screen-space
roughness or invent another distance curve. Design a separately proven object
material mapping:

- retain normal-map alpha as specular amplitude;
- map engine `glossPower` to GGX roughness;
- keep dielectric `F0` unless a real per-material metalness mask exists;
- bound direct specular energy consistently with the vanilla HDR path;
- decide explicitly whether and how native specular fade applies to that new
  lobe;
- validate ordinary, projected-shadow, high-light, and only-specular rows
  independently.

That would be a new engine/material contract, not an NVR source port, and must
be labeled accordingly.

Implementation status (2026-07-16): complete in source after the lighting-transition ownership audit closed the native row handoff. OMV now uses normal alpha as specular amplitude, engine gloss power as the bounded lobe-width control, dielectric Schlick Fresnel, and the proven native fade only on combined-row specular. The direct diffuse equation is identical on both sides of the row transition. The implementation deliberately uses a normalized Blinn distribution instead of reapplying the native fade to NVR's unbounded GGX peak.

## Do Not Repeat

- Do not tune another `smoothstep` around `LightData[0].w` before fixing the
  roughness/material model.
- Do not apply a vanilla fade scalar to a materially different, unbounded lobe
  merely because the interpolator is available.
- Do not enable `ddx`/`ddy` roughness changes globally without temporal tests.
- Do not treat an unused NVR object metallicness constant as a metalness map.
- Do not share object and terrain material assumptions.
- Do not globally disable PBR or reduce object coverage as the fix.
- Do not require a diagnostic-only gameplay run after the static contracts are
  executable tests. Detailed telemetry remains an opt-in fallback, not an
  acceptance dependency.

## 2026-07-19 Static Closure

The later audit found three additional defects beyond the original GGX/fade
hybrid:

- the hard zero half-vector fix still stepped at its epsilon boundary, so it was
  replaced by branchless soft normalization and a numerical camera sweep;
- only-light and diffuse-point replacements had dropped vanilla's attenuation
  lookup sampler/interpolator ABI, now restored exactly from package-19
  disassembly;
- high-light pixel rows evaluated inactive slots, allowing zero-radius NaNs and
  wasting BRDF work, now prevented by finite attenuation and uniform branches.

The static suite now compiles every registered variant, verifies the complete
source-derived row list and implemented pair ABI, checks exact special sampler
masks, sweeps the bounded BRDF and fade, and enforces bytecode, instruction, and
texture-sample budgets for every object shader. Normal runtime mode also avoids
per-draw diagnostic work and page-query pointer validation.
