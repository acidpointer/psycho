Overview

  libpsycho is a Windows-only (x86/x64) hooking library for plugin development with three hook types: Inline (JMP), IAT, and VMT hooks. The codebase demonstrates solid understanding of low-level Windows programming with strong safety considerations.

  Critical Issues

  1. Unsafe Memory Access in PE Parser (libpsycho/src/os/windows/pe.rs:53-56)

  Severity: HIGH
  let pe_bytes = unsafe { std::slice::from_raw_parts(
      module_base as *const u8,
      65536  // Hardcoded size!
  ).to_vec() };
  Issues:
  - Hardcoded 65536 bytes read without validating actual module size
  - No bounds checking - can read beyond module boundaries
  - Can trigger access violations if module < 64KB or near address space boundary

  Recommendation:
  // Get actual module size from PE headers or VirtualQuery
  let dos_header = unsafe { *(module_base as *const IMAGE_DOS_HEADER) };
  let nt_headers = unsafe { module_base.add(dos_header.e_lfanew as usize) as *const IMAGE_NT_HEADERS };
  let size_of_image = nt_headers.OptionalHeader.SizeOfImage;
  let pe_bytes = unsafe { std::slice::from_raw_parts(module_base as *const u8, size_of_image as usize).to_vec() };

  2. Thread Safety Issue in FFIRef (libpsycho/src/ffi/ref.rs:61-68)

  Severity: MEDIUM
  impl<T> AsRef<T> for FFIRef<T> {
      fn as_ref(&self) -> &T {
          let ptr = self.as_ptr();
          unsafe { ptr.as_ref().unwrap() }  // Shared reference without synchronization!
      }
  }
  Issues:
  - AsRef creates shared references from AtomicPtr without proper synchronization
  - Violates Rust's aliasing rules if pointer is updated concurrently
  - thread_id field suggests cross-thread usage, but no actual safety mechanism

  Recommendation:
  Either remove Send + Sync or use proper synchronization (RwLock/Mutex). Current design is unsound for concurrent access.

  3. Recursion Guard Leak (libpsycho/src/os/windows/hook/inline/inlinehook.rs:281-293)

  Severity: LOW-MEDIUM
  pub fn original(&self) -> InlineHookResult<F> {
      let _recursion_guard = RecursionGuard::try_new(self.id).ok_or_else(|| {
          log::error!("[{}] Recursive hook call detected", self.name);
          InlineHookError::RecursiveHook
      })?;

      unsafe { Ok(self.original_fn.as_fn()?) }  // Can return early, guard drops
  }
  Issue: If as_fn() returns Err, guard drops normally. But thread-local guard remains until function completes, preventing legitimate nested calls.

  Suggestion: Consider using a counter instead of HashSet to allow controlled recursion depth.

  ---
  Design Issues

  4. Inconsistent Memory Protection Restoration (libpsycho/src/os/windows/hook/inline/inlinehook.rs:235-260)

  In disable():
  let _old_protection = virtual_protect(
      self.target_address,
      PageProtectionFlags::PageExecuteReadWrite,
      trampoline_stolen_size,
  )?;

  // ... write stolen bytes ...

  virtual_protect(self.target_address, self.original_protection, trampoline_stolen_size)?;

  The code restores original_protection instead of the value returned from the first virtual_protect(). While this is likely correct (original protection is cached during creation), it could fail if protection changed externally.

  Recommendation: Document why original_protection is safe here or use returned _old_protection.

  5. Trampoline Jump Verification Logic Error (libpsycho/src/os/windows/hook/inline/trampoline.rs:89-92)

  let jump_back = create_jump_bytes(jump_back_offset, return_address)?;

  // Verify jump back is correct
  create_jump_bytes(jump_back.as_ptr() as *mut c_void, jump_back_offset)?;  // Wrong direction!

  This verifies a jump FROM the jump bytes TO the trampoline offset, which is backwards. Should verify the actual jump_back bytes.

  Fix:
  verify_jump_bytes(&jump_back, jump_back_offset, return_address)?;

  6. Missing Drop Implementation (libpsycho/src/os/windows/hook/iat/iathook.rs, vmthook.rs)

  IAT and VMT hooks don't implement Drop to disable hooks on cleanup, unlike InlineHook. This can leave hooks active after the hook object is destroyed.

  Recommendation: Implement Drop trait for consistency:
  impl<F: Copy + 'static> Drop for IatHook<F> {
      fn drop(&mut self) {
          if self.is_enabled() {
              let _ = self.disable();
          }
      }
  }

  ---
  Code Quality Issues

  7. FnPtr Alignment Check (libpsycho/src/ffi/fnptr.rs:60-63)

  if (raw_ptr as usize) % 2 != 0 {
      error!("Function pointer has invalid alignment: {:p}", raw_ptr);
      return Err(FnPtrError::FunctionPtrAlign);
  }

  Checks for 2-byte alignment, but x86/x64 function pointers should typically be 4/8 byte aligned or at minimum 1-byte (any alignment is technically valid). The % 2 check is odd.

  Recommendation: Use std::mem::align_of::<*mut c_void>() or remove check entirely (functions can be unaligned on x86).

  8. EventEmitter on() Off-by-One (libpsycho/src/common/emitter.rs:72-91)

  pub fn on<F: Fn(&P) + Send + Sync + 'a>(&self, event: E, callback: F) -> ListenerId {
      let mut listener_id = self.last_id.write();
      let listener = Listener::new(*listener_id, callback);

      // ... insert listener ...

      *listener_id += 1;
      *listener_id  // Returns NEXT id, not the one used!
  }

  Returns the incremented ID instead of the ID actually assigned to the listener.

  Fix:
  let id = *listener_id;
  let listener = Listener::new(id, callback);
  // ... insert ...
  *listener_id += 1;
  id  // Return actual ID

  9. Magic Numbers and Hardcoded Constants

  - MAX_VMT_SIZE: usize = 1024 (vmthook.rs:31) - No justification for this limit
  - MAX_INSTRUCTIONS: usize = 20 (inline/utils.rs:266) - Arbitrary
  - MAX_STEAL_SIZE: usize = 64 (inline/utils.rs:267) - Should be documented
  - 65536 (pe.rs:55) - Critical hardcoded value (see issue #1)

  Recommendation: Move to constants module with documentation explaining rationale.

  10. Unused VMT Utils Module (libpsycho/src/os/windows/hook/vmt/utils.rs)

  Contains unused imports and no actual utilities. Should be removed or implemented.

  ---
  Safety Concerns

  11. Transmute Usage in FnPtr (libpsycho/src/ffi/fnptr.rs:93-95, 130-132)

  let addr = unsafe { std::mem::transmute_copy::<T, usize>(&function) };
  let result: T = unsafe { std::mem::transmute_copy::<usize, T>(&addr) };

  Uses transmute_copy which can cause double-drop if T is not Copy. The Copy bound exists, but this is still fragile.

  Recommendation: Use std::mem::transmute directly with size assertions or document why transmute_copy is necessary.

  12. Unchecked Pointer Arithmetic (Multiple locations)

  Throughout the codebase, pointer arithmetic like target.add(offset) is used without overflow checks. While typically safe in practice, could benefit from checked arithmetic in debug builds.

  Example (inline/utils.rs:86):
  let return_address = unsafe { target.add(jump_size) };  // No overflow check

  Positive Aspects

  ✅ Excellent Error Handling

  - Consistent use of thiserror for error types
  - Detailed error messages with context
  - Proper error propagation throughout

  ✅ Strong Documentation

  - Good module-level docs
  - Inline comments explaining complex logic (especially in create_jump_bytes)
  - Safety requirements clearly documented

  ✅ Comprehensive Logging

  - Extensive use of log macros at appropriate levels
  - Debug messages aid troubleshooting
  - Trace-level logging for detailed execution flow

  ✅ Smart Architecture Abstractions

  - Clean separation of concerns (hook trait, implementations, utilities)
  - Good use of RAII guards (AllocGuard, RecursionGuard, ScopedInlineHook)
  - Platform-specific code properly isolated

  ✅ Memory Safety Considerations

  - validate_memory_access() before operations
  - AllocGuard for cleanup on failure
  - Instruction cache flushing after code modification
  - Memory barriers for multi-threaded visibility

  Recommendations Summary

  High Priority

  1. Fix PE parser memory read - Critical safety issue
  2. Fix FFIRef thread safety - Soundness issue with Send/Sync
  3. Add Drop to IAT/VMT hooks - Prevent hook leaks
  4. Fix EventEmitter ID return - Correctness bug

  Medium Priority

  5. Fix trampoline verification logic
  6. Document/fix memory protection restoration strategy
  7. Add overflow checks in pointer arithmetic
  8. Move magic numbers to documented constants

  Low Priority

  9. Review FnPtr alignment check necessity
  10. Clean up unused code (vmt/utils.rs)
  11. Consider recursion counter vs. guard for InlineHook
  12. Add more assertions in unsafe blocks for debugging

  Overall Assessment

  Score: 7.5/10

  Strengths:
  - Well-structured, idiomatic Rust
  - Thoughtful safety considerations
  - Good error handling and logging
  - Clean abstractions

  Weaknesses:
  - Several safety issues requiring fixes
  - Inconsistent patterns across hook types
  - Some unclear design decisions
  - Critical hardcoded values without validation

  The codebase shows strong fundamentals but needs attention to the identified safety issues, particularly in PE parsing and FFI reference handling, before production use.
